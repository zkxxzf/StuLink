# 宿舍分配算法设计规范

> **版本**: v4.1  
> **最后更新**: 2026-7-5  
> **基于**: 宿管老师实际分配经验 + 两阶段估算 + 7种情形逐级尝试

---

## 一、核心原则

1. **合班仅在房间不足时发生**：房间充足时不合班，每个班独立房间
2. **合班硬约束**：同性别、同年级、同班型（强基/卓越分开），≤6人/间；选科方向不同可合班
3. **性别严格隔离**：男女分配独立计算、互不影响
4. **均衡优先**：同一班级同性别各房间人数尽量平均
5. **同班级同楼层房间连续**：同一个班级的宿舍尽量安排在同一楼栋、同一楼层、房间号连续的宿舍
6. **合班宿舍居中**：如果设置合班宿舍，合班宿舍的房间号尽量在两个班级宿舍的中间位置
7. **先算房间数量，后排房间顺序**：先确定所需的房间总数和方案，再按顺序分配具体房间号
8. 本算法只计算哪个宿舍分给哪个班级，哪个班级住在哪个宿舍，不涉及具体床位分配给具体学生的计算

---

## 二、两阶段计算流程

### 概念说明

- **6人间**：物理床位6个，无论如何入住上限为6人
- **8人间**：物理床位8个，根据情形上限可为6、7或8人
- **合班宿舍**：两个或以上班级共用的宿舍，无论如何上限为6人（仅情形7上限为8人）

### Phase 1：选班后（第1步完成时）立即计算

**时机**：用户在Step1选择完年级、班级、性别后。

**计算内容**：

```
以男生为例（女生同理）：

1. 按班级统计住校男生人数
2. 最多所需房间数 = sum(ceil(每班男生人数 / 6))
   - 这是按每个班都住6人间、不合班的最坏情况估算
   - 实际分配所需的房间数绝不会超过这个值
3. 最少所需床位数 = 总住校男生人数
```

**用途**：
- 最多所需房间数指导Step2选房：所选房间数应≥这个值才"可能"装下所有人
- 最少所需床位数指导Step2选房：所选床位总数应≥这个值才"可能"装下所有人
- 前端可以在Step2页面实时对比"已选房间数 vs 最多所需房间数"和"已选床位数 vs 最少所需床位数"

### Phase 2：选房后（第2步完成时）分情形计算

**时机**：用户在Step2选择完楼栋、楼层、房间后，已知：
- 男生可用6人间数量：`count_6_male`
- 男生可用8人间数量：`count_8_male`
- 女生同理

**7种情形按优先级从高到低**：

| 情形 | 6人间入住 | 8人间入住 | 合班 | 合班上限 | 说明 |
|------|----------|----------|------|---------|------|
| 情形1 | ≤6人 | ≤6人 | 无 | - | 最舒适：8人间也只住6人 |
| 情形2 | ≤6人 | ≤6人 | 有 | ≤6人 | 舒适+合班减少房间用量 |
| 情形3 | 6人 | 7人 | 无 | - | 8人间升级到7人 |
| 情形4 | 6人 | 7人 | 有 | ≤6人 | 7人+合班 |
| 情形5 | 6人 | 8人 | 无 | - | 8人间住满 |
| 情形6 | 6人 | 8人 | 有 | ≤6人 | 8人间住满+合班 |
| 情形7 | 6人 | 8人 | 有 | ≤8人 | 极限模式：合班也可住满 |

**逐级尝试**：从情形1开始，依次尝试每种情形。对每个情形，判断（以男生为例）：
- 该情形下总有效容量 = `count_6_male × 6人入住上限 + count_8_male × 8人入住上限` 
- 如果总有效容量 ≥ 总男生人数 → **可能可行**，进入分配算法
- 如果总有效容量 < 总男生人数 → 直接跳下一个情形
- 找到第一个能装下所有人的情形，即停止，使用该情形进行分配

对于情形2、4、6、7（有合班），如果通过合班能使总人数满足，则选择该情形。

## 三、各情形详细计算（以单性别男生为例）

### 通用变量

```
students = 男生总人数
rooms_6 = 可用6人间数量   (count_6_male)
rooms_8 = 可用8人间数量   (count_8_male)
total_rooms = rooms_6 + rooms_8
```

### 情形1：均≤6人，无合班

```
6人间入住上限 = 6
8人间入住上限 = 6
总有效容量 = rooms_6 × 6 + rooms_8 × 6 = total_rooms × 6

条件：总有效容量 ≥ students  （且能分配到每个班）

最简判断：
  sum(ceil(各班人数/6)) ≤ total_rooms   （每个班按6人间分配）

分配方式：
  每个班分配 ceil(人数/6) 间，班内均衡分配
  可有多余空房间
```

### 情形2：均≤6人，有合班

```
6人间入住上限 = 6
8人间入住上限 = 6
合班上限 = 6

条件：情形1中 sum(ceil/6) > total_rooms, 但合班可解决碎片

合班计算：
  1. 先按情形1分配，收集各班级剩余未分配学生
  2. 同合班组（同年级+同班型+同方向）内尝试配对
  3. 多班剩余之和 ≤ 6 即可合并到一间
  4. 合班后总算所需房间数 ≤ total_rooms → 可行

可合班数量范围：取决于具体班级剩余分布
最低需房间数 = ceil(students / 6)  （完全密集打包）
最高需房间数 = sum(ceil/6)        （不合班）
```

### 情形3：6人=6, 8人=7，无合班

```
6人间入住上限 = 6
8人间入住上限 = 7
总有效容量 = rooms_6 × 6 + rooms_8 × 7

条件：总有效容量 ≥ students

分配方式：
  每个班按 人数/(6或7) 的比例分配房间
  8人间尽量分给人数多的班级
```

### 情形4：6人=6, 8人=7，有合班

```
6人间入住上限 = 6
8人间入住上限 = 7
合班上限 = 6

条件：情形3中仍有未分配学生，通过合班可解决
计算方式与情形2类似，但8人间按7人计算
```

### 情形5：6人=6, 8人=8，无合班

```
6人间入住上限 = 6
8人间入住上限 = 8
总有效容量 = rooms_6 × 6 + rooms_8 × 8

条件：总有效容量 ≥ students
此时是物理床位满打满算的情况
```

### 情形6：6人=6, 8人=8，有合班（合班≤6）

```
6人间入住上限 = 6
8人间入住上限 = 8
合班上限 = 6

条件：情形5中仍有未分配学生
注意：此情形下床位一定是充足的（否则会在情形5前就被拦截），
      问题出在班级碎片化，通过合班可解决
```

### 情形7：6人=6, 8人=8，有合班（合班≤8）

```
6人间入住上限 = 6
8人间入住上限 = 8
合班上限 = 8  （极限模式）

条件：情形6中仍有未分配学生且用户确认使用极限模式
用此情形前应弹窗让用户确认：
  "所有8人间和合班宿舍均已住满，确认按此方案执行？"
```

## 四、房间分配算法（确定情形后）

### 分配原则

1. 房间先按 **(楼栋, 楼层, 房间号)** 排序
2. **同班级的房间必须同楼层连续**：同一班级的多个房间尽量在同一楼层内，房间号连续
3. 合班宿舍放在参与合班的两个（或多个）班级房间的中间位置
4. 每个班级内各房间学生人数均衡分配

### 分配步骤

```
第1步：按(楼栋, 楼层)分组房间，每组内按房号排序

第2步：按班级人数降序排列班级（大班优先选连续房间）

第3步：对每个班级：
  a. 确定该班在所选情形下需要的房间数
  b. 在合适楼层找到连续的空闲房间段
  c. 分配房间
  d. 在班内均衡分配学生人数

第4步：如有合班宿舍：
  a. 合班宿舍选两个班级房间之间的位置
  b. 标记 combined_class = '合班'
```

### 班内均衡分配公式

```
students = 班级学生数
rooms = 分配给该班的房间数

base = students // rooms        # 每间基础人数
remainder = students % rooms    # 前remainder间各+1人

for i, room in enumerate(rooms):
    expected = base + (1 if i < remainder else 0)
    if expected > room.capacity:
        expected = room.capacity  # 不超过房间物理床位
```

## 五、合班约束

合班宿舍必须同时满足：

| 约束 | 规则 |
|------|------|
| 性别 | 同性别 |
| 年级 | 同年级 |
| 班型 | 同班型（强基班/卓越班不能混） |
| 人数 | 参与合班的所有班级剩余人数之和 ≤ 合班上限 |
| 参与次数 | 每个班级最多参与1个合班 |

合班分组 key：
```
group_key = f"{grade}|{class_type}"
只有 group_key 相同的班级才能合班
```

没有 ClassProfile 记录的班级 → `class_type=None`，自成一组。

**说明**：选科方向不同的班级可以合班，只要满足同性别、同年级、同班型条件即可。

## 六、前端交互流程

### Step1：选择班级
- 三维选择：年级 + 班级 + 性别
- 快捷操作：「男生全选」「女生全选」
- 实时显示每班住校人数 + 总人数
- **选完后计算并显示**：最多所需房间数、最少所需床位数

### Step2：选择房间
- 按楼栋→楼层→房间三级展示
- 每间显示：楼栋、楼层、房号、性别、类型（6/8人间）、床位数、占用状态
- 已分配房间黄色背景标记
- **实时对比**：
  - 已选房间数 vs 最多所需房间数（来自Phase 1）
  - 已选床位数 vs 最少所需床位数（来自Phase 1）
- 床位不足时禁用「下一步」

### Step3：预览方案
- 按班级分组展示，男女分开
- 执行两阶段计算，返回分配方案
- 如果需要合班确认：弹窗列出合班组合
- 如果有已分配房间冲突：弹窗选择「保留」或「覆盖」
- 如果触发情形7（极限模式）：弹窗让用户确认

### Step4：执行写入
- 修改 Room 表的 grade / class_name / gender / combined_class
- 不修改 BedAssignment
- 记录操作日志

## 八、示例场景

### 场景A：情形1直接满足（2025级全选+5-6楼）

```
输入：2025级全选，东西楼5-6楼
男生：246人，10个班
女生：247人，10个班
房间：84间（男生42间、女生42间），612床位

Phase 1（选班后）：
  男生 sum(ceil/6) = 46间（最多需要）
  女生 sum(ceil/6) = 42间（最多需要）

Phase 2（选房后）：
  男生有44间可用 → sum(ceil/6)=46 > 44 → 情形1不满足

尝试情形2（≤6，合班）：
  ceil(246/6)=41间 ≤ 44间 ✓
  合班配对后所有学生可分配 → 情形2满足！
  使用情形2执行分配
```

### 场景B：床位充足但ceil碎片化（2024级+3-4楼）

```
输入：2024级全选，东西楼3-4楼
男生：250人，10个班
房间：男生44间（8间6人+36间8人），336床

Phase 1：男生 sum(ceil/6) = 46间

Phase 2：
  情形1：46间 > 44间 → 不满足
  情形2：ceil(250/6)=42间 ≤ 44间, 合班后可解决 → ✓
  
分配结果：83间使用，1间合班(01班+02班)，unassigned=0
```
